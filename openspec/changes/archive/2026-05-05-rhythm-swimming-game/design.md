## 上下文

本项目是一款全新的音乐节奏 × 游泳竞速混合型多人游戏。玩家在10人房间中，通过按照音乐节拍进行输入操作来控制角色的游泳动作，节奏准确度直接影响泳速和排名。项目从零开始构建，需要同时处理音乐节奏判定的精确性和多人实时竞技的同步性。

## 目标 / 非目标

**目标：**
- 设计一套将音乐节奏判定与游泳速度紧密耦合的核心玩法系统
- 实现10人实时竞技房间，确保公平性和低延迟体验
- 构建可扩展的节奏引擎，支持不同BPM和难度的曲目
- 提供流畅的游泳动画表现，让玩家操作反馈直观明确

**非目标：**
- 不实现单人剧情/关卡模式（首期仅做多人竞技）
- 不实现自定义曲目上传功能
- 不实现复杂的角色养成/装备系统
- 不处理跨平台（首期仅针对单一平台）

## 决策

### 1. 节奏判定采用客户端判定 + 服务端校验模式

**选择**：客户端本地进行节奏判定并立即反馈，同时将输入时间戳上报服务端做反作弊校验。

**替代方案**：
- 纯服务端判定：延迟过高，音游体验极差
- 纯客户端判定：容易作弊

**理由**：音乐节奏游戏对输入反馈延迟极为敏感（<50ms），纯服务端判定在网络环境下无法满足。采用客户端本地判定保证手感，服务端异步校验保证公平。

### 2. 泳速计算模型采用"节奏准确度 → 划水效率 → 速度"三层映射

**选择**：
- 第一层：输入时间与节拍时间差 → 判定等级（Perfect/Great/Good/Miss）
- 第二层：判定等级 → 划水效率系数（1.0 / 0.8 / 0.5 / 0.0）
- 第三层：划水效率 × 基础速率 × 连击加成 × 体力系数 = 实际泳速

**替代方案**：
- 直接将判定等级映射为固定距离推进：缺少连续性，手感生硬
- 物理模拟游泳：过于复杂且难以与节奏系统耦合

**理由**：三层映射既保留了清晰的因果关系（打准 → 游得快），又提供了足够的调参空间来平衡游戏节奏。

### 3. 房间同步采用帧同步 + 状态快照混合方案

**选择**：比赛中使用固定帧率（60fps逻辑帧）帧同步推进，每隔一定帧数服务端广播状态快照用于校正。

**替代方案**：
- 纯状态同步：带宽消耗大，10人场景压力高
- 纯帧同步：网络抖动容易导致卡顿

**理由**：帧同步保证所有客户端逻辑一致，状态快照用于纠错和断线重连。对于本游戏，核心逻辑（泳速计算）较轻量，帧同步开销可控。

### 4. 音乐数据预处理采用离线标注 + 运行时加载

**选择**：每首曲目在上线前通过离线工具进行BPM检测和节拍标注，生成节拍数据文件（JSON格式），运行时直接加载。

**替代方案**：
- 运行时实时BPM检测：计算开销大、准确度不稳定
- 手工标注：效率低但精度最高

**理由**：离线处理可以结合算法自动检测 + 人工校准，确保节拍数据准确可靠。运行时只需加载预处理好的数据，性能开销极小。

### 5. 体力系统采用线性消耗 + 节奏惩罚模型

**选择**：体力按时间线性消耗基础值；Miss判定额外扣除体力；体力低于阈值时速度系数降低。

**替代方案**：
- 无体力系统：缺少策略性，比赛后半段无变数
- 复杂的非线性体力模型：增加理解成本

**理由**：简单直观的体力系统为比赛增加策略维度（前半段是否"省力"），同时惩罚频繁Miss的玩家，强化"打准才能赢"的核心循环。

### 6. 冲刺段落采用曲目预标注触发

**选择**：在曲目节拍数据中预先标注"高潮段落"区间，进入该区间时自动触发冲刺模式，判定权重翻倍。

**替代方案**：
- 玩家手动触发冲刺：与节奏概念脱节
- 基于连击数触发：优势滚雪球，不利追赶

**理由**：与音乐高潮自然结合，所有玩家同时进入冲刺，保持公平竞争的同时增强音乐表现力。

## 风险 / 权衡

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 网络延迟导致节奏判定不公平 | 高延迟玩家天然劣势 | 客户端本地判定 + 服务端容忍窗口（±网络延迟补偿） |
| 10人同步带宽压力 | 可能导致卡顿 | 精简同步数据包，仅同步位置和关键状态变化 |
| 音乐版权问题 | 上线阻塞 | 优先使用原创/免版权音乐，预留版权音乐接入接口 |
| 作弊（变速外挂） | 破坏公平性 | 服务端校验输入时间戳合理性，异常检测后踢出 |
| 体力系统可能降低乐趣 | 后半段"划水"体验差 | 通过数值调优确保体力不会过早耗尽，提供恢复机制 |
| 冲刺段差距过大翻盘困难 | 落后玩家失去动力 | 冲刺倍率上限控制 + 追赶机制（落后者获得微量加成） |
